Cách Quản Lý Dự Án Thông Qua Jira & Github

Phương pháp phối hợp Agile, phân công công việc và quản trị mã nguồn trong nhóm.

Theo dõi tiến độ bằng Jira Software

Nhóm thiết lập Sprint ngắn hạn 1 tuần trên Jira để quản lý backlog và phân tách nhiệm vụ rõ ràng:

  • AI Workstream: Xây dựng API /v1/triage, tích hợp Bedrock SDK, viết hợp đồng Telemetry và chẩn đoán RCA toán học.
  • DevOps Workstream: Bootstrap EKS Cluster sandbox, xây dựng module mạng VPC Link, cài đặt addons (External Secrets, OPA Gatekeeper, ArgoCD).
  • Integration Tests: Phối hợp kiểm thử E2E 3 kịch bản sự cố trên môi trường sandbox.
Quy trình Git Workflow & Peer Review

Áp dụng quy tắc bảo mật code nghiêm ngặt trước khi đưa vào cluster:

  • Nhánh main được bảo vệ (Protected Branch), cấm push trực tiếp.
  • Mọi tính năng được phát triển trên nhánh feature/*, yêu cầu tối thiểu **1 code review** từ thành viên khác trước khi Approve PR.
  • Gắn tag tự động theo Git Commit SHA động giúp tránh xung đột ECR image.

Tại Sao Hạ Tầng Này Phù Hợp Để Triển Khai AI?

Các yếu tố quyết định để chạy một AIOps Engine bền bỉ và bảo mật trên EKS.

Đặc trưng tải xử lý của AI Engine

AIOps Engine không phải là dịch vụ chạy LLM liên tục (high-throughput), mà hoạt động theo dạng **sự kiện đột xuất** (event-driven). Khi sự cố (Alert) xảy ra, hệ thống cần tài nguyên tức thời để:

  • Truy vấn hàng ngàn dòng logs/metrics nội bộ (Compute-intensive).
  • Gọi Bedrock API phân tích suy luận (I/O-intensive).
  • Đòi hỏi bảo mật tuyệt đối, ngăn rò rỉ dữ liệu log khách hàng ra internet.
Kết nối riêng tư (VPC Private Link)
Gọi Bedrock API không qua internet bằng SigV4 VPC Interface Endpoints. Đảm bảo dữ liệu nhạy cảm không rời mạng nội bộ.
Định tuyến Hybrid Mode thông minh
Tự động chuyển đổi giữa chẩn đoán toán học cục bộ (Deterministic RCA) và AI Agent dựa trên độ phức tạp, giúp tối ưu hóa chi phí token và thời gian phản hồi.

Kiến Trúc Hạ Tầng - Khả Năng Đáp Ứng

Bảo đảm độ sẵn sàng (Availability) và khả năng sống sót cao của dịch vụ trong vùng mạng riêng tư.

Đặc Tính Đáp Ứng Dịch Vụ (Availability)

  • Thiết kế dự phòng 2 Replicas: Deploy tối thiểu 2 pods cho tf1-api phân tán trên các Node khác nhau.
  • Pod Disruption Budget (PDB): Enforce cấu hình minAvailable: 1 để Kubernetes luôn giữ ít nhất 1 Pod API hoạt động trong quá trình nâng cấp cluster hoặc bảo trì node.
  • DynamoDB Session Caching: Lưu vết trạng thái chẩn đoán sự cố, giúp khôi phục tức thời trạng thái xử lý nếu pod bị sập giữa chừng (Stateless API design).
  • Internal Load Balancer: Nhận request cục bộ qua Internal ALB, cấu hình health check chặt chẽ cổng 8080.

Kiến Trúc Hạ Tầng - Khả Năng Mở Rộng

Cơ chế scale-out hạ tầng tự động để đối phó với cơn bão alert (Alert Storm).

Cơ Chế Scale Đa Lớp (Multi-tier Autoscaling)

1. HPA (Horizontal Pod Autoscaler)

Tự động nhân bản Pod tf1-api từ 2 lên tối đa 10 Pod dựa trên tải CPU sử dụng (Ngưỡng kích hoạt: 70% CPU).

2. SQS Queue-depth Scaling

Pod tf1-worker (SQS consumer) tự động scale số lượng bản sao dựa trên độ dài hàng đợi tin nhắn của AWS SQS, giải phóng nhanh nghẽn cảnh báo khi có biến cố lớn.

3. Kubernetes Cluster Autoscaler

Khi pods rơi vào trạng thái Pending do thiếu hụt tài nguyên CPU/Mem của máy chủ, Cluster Autoscaler tự động scale số lượng EC2 Node (Min: 2, Max: 10).

Cấu hình HPA thực tế (Sandbox vs Prod Overlay)

Sử dụng Kustomize Overlays để phân tách cấu hình Autoscaling theo môi trường nhằm kiểm soát ngân sách AWS:

# patches/hpa-patch.yaml spec: minReplicas: 2 maxReplicas: 5 # Sandbox (Prod: 10) metrics: - type: Resource resource: name: cpu target: type: Utilization averageUtilization: 70

Kiến Trúc An Ninh & Bảo Mật Chuỗi Cung Ứng

Các lớp phòng thủ từ CI/CD cho tới thời điểm runtime trong Cluster.

1. Bảo mật mạng (Network Isolation)
EKS Nodes chạy trong Private Subnets, hoàn toàn không gán IP Public. Bật các Interface VPC Endpoints cho S3, DynamoDB, Bedrock, Secrets Manager để các pod kết nối nội bộ thông qua AWS PrivateLink bảo mật.
2. Xác thực định danh (Auth & IAM)
Áp dụng phân quyền nguyên tắc đặc quyền tối thiểu (Least Privilege) bằng EKS IRSA (IAM Roles for Service Accounts). Toàn bộ API calls sang Bedrock được ký mã hóa SigV4. API endpoint xác thực qua SERVICE_AUTH_TOKEN (Bearer token).
3. Tính toàn vẹn (Integrity)
Sử dụng Cosign ký số keyless cho container image qua GitHub OIDC. Enforce chính sách Sigstore Policy Controller trong cluster và OPA Gatekeeper cấm quyền root của Pod (PSS Restricted).

Observability & Khả Năng Quan Sát

Đo lường sức khỏe dịch vụ và giám sát hoạt động của AI Engine.

  • aiops_llm_estimated_cost_usd_total: Chi phí USD thời gian thực.
  • aiops_degraded_mode_total: Số lần AI engine lỗi phải fallback sang deterministic.
  • aiops_triage_request_duration_seconds: Đo lường độ trễ API.

Kiến Trúc Vận Hành & An Ninh AI Engine

Bảo mật chuỗi cung ứng hình ảnh, tự động hóa Canary Rollouts và cơ chế Autoscaling đa lớp.

Quy trình Triển khai & An ninh

  • Bảo mật chuỗi cung ứng (Sigstore): Đóng gói Dockerfile Multi-stage non-root. Trivy quét chặn cứng lỗi HIGH/CRITICAL. Ký Cosign keyless. EKS Admission Controller (Sigstore policy) chặn đứng việc chạy pod nếu image chưa được ký hợp lệ.
  • Canary Rollouts tự động (Argo Rollouts): Đóng gói qua ArgoCD GitOps. Dịch vụ tf1-api được deploy Canary (10% → 50% → 100%). Quy trình AnalysisRun giám sát liên tục p99 latency (<800ms) và error-rate 5xx (<1%) qua Prometheus để tự động Rollback nếu xảy ra sự cố.
  • Bảo mật Tầng Admission (7 Gatekeeper Policies): Triển khai vượt mức baseline thông thường:
    • Siết security context toàn diện, chặn cứng hostNetwork/hostPID.
    • Chỉ cho phép kéo image từ Registry của dự án.
    • Bắt buộc khai báo cả requests và limits tài nguyên để chống chiếm dụng khi dùng chung node (Noisy Neighbor).
    • Referential constraint: Bắt buộc mọi Namespace mới phải có sẵn NetworkPolicy.
    • Kỷ luật triển khai: Chạy chế độ audit-only trước để kiểm tra thực tế, sau đó mới bật chặn cứng (enforce) để tránh gây outage bất ngờ.

Mở rộng Hạ tầng & Dữ liệu

  • Autoscaling 3 lớp độc lập:
    • tf1-api: Scale theo CPU (70% HPA) và custom metric số lượng request/pod từ 2 đến 10 pods.
    • tf1-worker: Scale bằng KEDA dựa trên độ sâu hàng đợi SQS (backlog depth: 5 messages/pod), vượt qua giới hạn của HPA thông thường khi có bão alert.
    • Node EC2: Cluster Autoscaler tự thêm node khi pod rơi vào trạng thái Pending.
  • Kết nối VPC PrivateLink & NAT-less: Pod chạy hoàn toàn trong Private Subnets, giao tiếp với Bedrock, DynamoDB, ECR qua VPC Endpoints (AWS PrivateLink). Loại bỏ hoàn toàn NAT Gateway giúp bảo mật tuyệt đối dữ liệu và tối ưu chi phí hạ tầng mạng.
  • Cô lập dữ liệu tenant: DynamoDB single-table lưu audit, state và cache phân công SME. Cách ly tenant triệt để bằng cách nhúng tenant_id ngay trong Partition Key.

Ingestion (Thu nhận Cảnh báo)

Idempotency Cache (Tối ưu hóa Chi phí)

4

AI Inference (Suy luận Bedrock LLM)

Prompt Engine kết hợp dữ liệu gửi qua VPC PrivateLink lên Bedrock Claude 3.5 Sonnet để phân tích nguyên nhân gốc (RCA).

5

Dispatch & Assign (Tự động Phân công)

Ghi log audit vào DynamoDB, đẩy báo cáo tương tác lên Slack Block Kit và tự động tạo ticket Jira gán cho đúng SME.

CI/CD Pipeline & Quy Trình GitOps

Vận hành quy trình đóng gói, rà soát mã nguồn tự động và đồng bộ ArgoCD.

Quy Trình Pipeline GitHub Actions

1. Build & Scan

Mỗi lượt push code lên Git, workflow tự động chạy test unit, build docker image và quét lỗ hổng thư viện bằng Trivy.

2. Ký Ảnh Đĩa Cosign

Pipeline sử dụng OIDC token để xác thực với Cosign và ký số cho ảnh đĩa (ECR image) một cách an toàn không cần lưu giữ key.

3. Đẩy Commit SHA Tag

Sử dụng Git Commit SHA ngắn làm tag cho image (e.g. :git-ac21f60). Khắc phục triệt để lỗi ghi đè tag cũ do chính sách ECR Immutable Tags.

ArgoCD GitOps Synchronizer

ArgoCD liên tục theo dõi repository Git chứa các file manifest. Khi phát hiện tag image mới cập nhật trên Kustomization.yaml, ArgoCD tự động thực hiện deploy đồng bộ (Sync) kéo container image mới nhất về chạy trên cluster.
